看完 Anthropic CCC 的体会

Date
February 6, 2026
Tags
  • ai
  • /compiler
  • /programming language

昨天读了 Anthropic 的《Building a C compiler with a team of parallel Claudes》。 他们让多个 Claude 并行写一个 C 编译器，和我最近在做的多 Agent 协作很接近，所以我重点看了任务怎么拆、代码怎么同步、结果怎么验收。

多个 agent 同时改一套代码，碰到的还是熟悉的软件工程问题。

读完后，我对 merge conflict 的容忍度高了一些。人看到上百个冲突，大概率已经傻眼了，agent 至少不会因为我制造冲突而恨我（也许）。只要 review 和回归测试能兜住，冲突的成本会降低许多。

测试也可以重新分层。每个 agent 未必需要在自己的任务上跑完整测试集。人类协作时，我们坚持全量测试通过再合并，很大一部分原因是担心把排查成本甩给别人。agent 碰到回归，通常会继续定位，甚至顺手修掉。

完整回归可以滞后。我以前给 Rust repo 提 PR 时见过类似流程。单个 PR 只跑部分测试，maintainer 定期合并一批 PR，再跑更完整的回归。这背后的权衡也值得借鉴。

整篇文章里最亮眼的，莫过于作者的测试框架，我觉得这部分最考验工程能力。作者用两周让编译器把 Linux 6.9 编译通过。模型负责继续写代码，测试框架负责暴露错误并缩小排查范围。

项目后期有点“按下葫芦起了瓢”，项目先告一段落了。
